너는 제조업 공급망 ESG/PCF 관리 SaaS **AIFIX 협력사 포털**의 시니어 프로덕트 디자이너다.

이번 작업 범위는 “협력사 포털 > 프로젝트 선택 > 1차 역할 프로젝트 진입 후(네비게이션2)” 화면 중
**[PCF 산정·전송] 탭**과 **[이력] 탭** 2개만 구체적으로 설계하는 것이다.
(비1차 화면은 만들지 말 것)

[고정 전제]
- 협력사 포털은 단일 원청 전용이다.
- 협력사는 구매/PCF 직무를 분리하지 않고 “통합 템플릿” 하나에 입력한다.
- 현재 사용자는 “1차 협력사”이며,
  하위차사 구조/데이터 조회 가능 + 하위 데이터 직접 수정 가능 + 최종 취합 후 원청 제출 책임이 있다.
- 핵심 흐름:
  데이터 관리(입력/수정, 계산 없음) → PCF 산정·전송(계산/검토/전송) → 데이터 조회(입력/결과 확인, 분리)
- PCF 산정·전송 탭에서 생성된 계산 결과는 “버전”으로 저장되고,
  이후 데이터 조회(PCF 결과 조회) 및 이력 탭에서 추적 가능해야 한다.

[공통 레이아웃]
- 상단에 프로젝트 컨텍스트 헤더를 고정 표시(모든 탭 공통):
  프로젝트명 / 내 차수(1차) / 제출 대상(원청사명) / 계약번호 / 마감일 / 상태
- 네비게이션2 탭바 유지:
  대시보드 / 공급망 관리 / 데이터 관리 / 데이터 조회 / PCF 산정·전송 / 이력
- 디자인 톤: B2B SaaS, 신뢰감, 정돈된 카드/테이블, 충분한 여백

=====================================================
A) [PCF 산정·전송] 탭 (1차 최종 취합/제출) 구체 설계
=====================================================
목표: 1차 협력사가 “전체 취합 결과”를 계산하고, 누락/미제출 리스크를 점검한 뒤
원청에 전송할 수 있게 한다. 이 탭은 ‘계산 실행 + 결과 검토 + 전송’이 한 흐름으로 이어져야 한다.

1) 상단 “계산/제출 메타 정보” 고정 패널(헤더 아래)
- 현재 역할: 1차
- 제출 대상: 원청사
- 계산 기준 버전(기준 데이터 버전/계산 버전 표시)
- 마지막 계산 시각
- 마지막 전송 시각
- 상태 배지: 계산 필요 / 계산 완료 / 전송 대기 / 전송 완료 / 재전송 필요
※ 사용자가 “내가 누구에게 무엇을 보내는지”를 헷갈리지 않게 한다.

2) 3-컬럼 또는 2-영역 레이아웃으로 구성(권장: 좌측 ‘점검/액션’ + 우측 ‘결과/분석’)
- 좌측: 전송 전 점검(체크리스트/경고) + 실행 버튼 영역
- 우측: PCF 결과 요약 + 상세 분석 영역

3) 좌측 영역: “전송 전 점검(Pre-submit Checklist)” 카드
체크 항목은 최소 아래를 포함:
- 미제출 하위차사 존재 여부
- 누락 데이터(필수값 미입력/검증 실패) 존재 여부
- 하위 반영 상태(반영률 % 또는 반영 완료/미완료)
- 계산 실패/경고 로그 존재 여부
각 항목은 ‘통과/주의/실패’ 상태로 표시하고,
실패/주의인 경우 바로 이동 CTA를 제공:
- 미제출 업체 보기(공급망 관리로 이동 or 필터 적용)
- 누락 항목 확인(데이터 관리로 이동 or 검증 실패 필터)
- 경고 상세 보기(계산 로그 패널로 스크롤)

4) 좌측 하단: “계산 실행/재계산” 액션 카드
버튼 2개를 명확히 구분:
- 계산 실행 (초기 계산)
- 재계산 (입력값 변경 후)
버튼 아래에는 계산 실행 시 생성되는 산출물 안내:
- “계산 결과는 버전으로 저장되며, 데이터 조회/이력에서 추적 가능합니다.”
※ 이 탭에서만 계산 버튼이 존재해야 하며, 다른 탭에 분산시키지 말 것.

5) 우측 상단: “PCF 결과 요약” 카드(가장 눈에 띄게)
- 총 PCF (큰 숫자)
- 단위(예: kgCO2e / 제품 단위)
- 계산 버전
- 하위 반영률(%) 또는 반영 완료 여부
- 주요 경고 수(있다면 배지로 표시)
※ 단, 세부 계산식/수학은 UI에 넣지 말고 결과 요약 중심

6) 우측 중단: “기여도(Contribution)” 요약 영역(차트/리스트 중 택1)
- 하위차사 기여도 Top N
- 활동데이터/구매데이터/배출계수 등 카테고리별 기여도
※ 색상 지정은 피하고, 구조만 제시

7) 우측 하단: “계산 로그/경고” 패널
- 경고(누락, 이상치, 반영 실패 등) 리스트
- 각 경고는 ‘원인/영향/권장 조치’ 3요소로 짧게 구성
- “조치하러 가기” 링크 제공(데이터 관리/공급망 관리로 이동)

8) 최하단(고정 또는 우측 상단 우선): “원청 전송” CTA 영역
- Primary 버튼: 원청 전송
- Secondary: 전송 미리보기(스냅샷 요약 모달) / 전송 취소(필요 시)
전송 버튼은 아래 조건에서 비활성 또는 경고 모달을 띄운다:
- 미제출 하위차사가 존재하는 경우
- 검증 실패/필수값 누락이 존재하는 경우
전송 완료 시:
- “전송 완료 배너” + 전송본 버전/시각 표시
- 재전송 버튼(조건 변경 시) 노출

[PCF 산정·전송 탭 금지사항]
- 입력 폼(통합 템플릿)을 이 탭에 넣지 말 것(입력은 데이터 관리 탭)
- 조회 탭처럼 원천 데이터 테이블을 길게 나열하지 말 것(여긴 ‘결과/검토/전송’ 탭)

=====================================================
B) [이력] 탭 (감사추적/Audit Trail) 구체 설계
=====================================================
목표: 1차 협력사가 프로젝트 내 변경·요청·계산·전송 기록을
“누가/언제/무엇을” 했는지 감사추적 형태로 확인할 수 있게 한다.
이력은 ‘증빙/실사 대응’이 목적이므로 신뢰감 있는 로그 UI로 설계한다.

1) 상단: “이력 필터 바” (테이블 위)
필터 최소 구성:
- 대상 범위: 전체/자사/하위차사(업체 선택)
- 차수: 전체/2차/3차
- 이벤트 타입: 입력변경/하위 수정/보완요청/재요청/계산/전송
- 기간(날짜 범위)
- 버전(계산 버전/전송 버전 선택)
- 사용자(수정자/요청자)
검색창(키워드): 업체명/계약번호/항목명 검색

2) 메인: “이력 테이블(로그)” 영역
테이블 컬럼 최소:
- 시각
- 이벤트 타입(배지)
- 대상(업체명/차수)
- 요약(무엇이 바뀌었는지 한 줄)
- 사용자
- 버전(해당 시)
- 링크(상세보기)
행은 최신순 정렬이 기본

3) 우측 또는 행 확장: “상세 패널” 제공(둘 중 하나로)
행 클릭 시 상세 패널에 다음을 보여준다:
- 이벤트 상세(변경 전/후 요약, 변경 항목 수)
- 관련 객체 링크: 해당 데이터(데이터 조회) / 해당 업체(공급망 관리) / 해당 계산 결과(PCF 산정·전송)
- 코멘트/사유(보완요청 메시지 등) 영역

4) “버전 비교” 기능(이력 탭에서 접근 가능)
- 계산 버전 2개를 선택 → 변경 요약(총 PCF 변화, 주요 기여도 변화, 경고 변화)만 표시
- 전송본(스냅샷) 선택 → 전송 당시 값 보기
※ 상세 수치 테이블을 길게 만들지 말고 ‘비교 결과 요약 카드’ 중심

5) “증빙/내보내기” 액션(선택적이지만 B2B SaaS 느낌 강화)
상단 우측에 다음 액션 버튼 중 1~2개 배치:
- 로그 내보내기(CSV/PDF) (버튼만, 상세 기능 구현은 하지 말 것)
- 전송본 보기(최근 전송 스냅샷)

[이력 탭 UX 규칙]
- 이력 탭은 수정/입력 기능이 없는 “조회 전용”이다.
- 모든 기록은 시간순/버전 기반으로 추적 가능해야 한다.
- ‘PCF 산정·전송’에서 생성된 계산/전송 이벤트가 이력에 반드시 남는다는 점을 UI에서 암시(아이콘/배지)

=====================================================
산출물
=====================================================
- 1차 프로젝트 진입 후 화면에서:
  1) PCF 산정·전송 탭 1개 화면(점검→계산→결과→전송 흐름이 한 화면에서 이어지도록)
  2) 이력 탭 1개 화면(필터 바 + 로그 테이블 + 상세 패널/확장 + 버전 비교 접근)
- 두 화면 모두 상단 프로젝트 컨텍스트 헤더와 네비게이션2 탭바 포함
- 세부 입력 필드/테이블 컬럼 정의는 최소로 유지하고, 구조/흐름/컴포넌트 배치 중심으로 설계할 것
